# use-fe — Unified Search Engine Frontend

`use-fe` is the React + TypeScript single-page application (SPA) for the **Unified Search Engine (USE)** platform. It provides a ChatGPT-style conversational UI that lets users ask questions in natural language and receive answers (with citations) generated by a Retrieval-Augmented Generation (RAG) backend over internal documentation and legal sources.

> The UI copy is in **Slovak** (e.g. "Ako vám môžem pomôcť?", "Nový chat", "Citácie"). Keep this localization in mind when editing or adding strings.

---

## 1. Purpose & Context

This frontend is one part of a larger multi-service solution (see the root `unified-search-engine.sln`). It talks to the backend `Use.Application.Service` (.NET) via a small set of REST endpoints and renders the conversational experience, including **persisted chat history**.

Sibling backend services in the workspace (for orientation only — the frontend does **not** call them directly):

| Service | Role |
| --- | --- |
| `Use.Application.Service` | Main API / orchestration layer (RAG, chat + history endpoints the FE consumes). |
| `Use.Indexing.Worker` | Crawls, parses, chunks, embeds and persists documents. |
| `Use.LlmService.Api` | LLM gateway (embeddings, generation). |
| `Use.Reranker.Service` | Python reranking microservice. |
| `use-qdrant-db` | Qdrant vector database (Docker). |
| `use-sql-db` | PostgreSQL database (Docker) — backs users, sessions, messages, RAG logs. |

The frontend only knows about the chat API exposed by `Use.Application.Service`.

## 4. Authentication

Auth is **no longer hard-coded**. It is provided by `src/auth` and consumed everywhere through the `useAuth()` hook, which exposes:

```ts
{ isAuthenticated, isLoading, user, mode, login, logout, configError }
```

`user` is `{ name, email, initials } | null`. The active implementation is selected by `VITE_AUTH_MODE`:

### `VITE_AUTH_MODE=dev` (default)
- No real identity provider. The user is "signed in" after a one-click **dev login** button on `/login`; the flag is persisted in `localStorage`, so a refresh keeps you signed in.
- Every API request carries dev identity headers that the backend understands (`CurrentUserService`): `X-User-Email`, `X-User-Name`, `X-Entra-Object-Id`, `X-Tenant-Id`.
- Values come from `VITE_DEV_*` env vars, defaulting to the backend's Development fallback user (`local.dev@use.local` / `Local Dev User` / `local-dev-user` / `local-dev-tenant`).

### `VITE_AUTH_MODE=msal` (Microsoft Entra ID)
- Uses `@azure/msal-browser` + `@azure/msal-react` with **redirect** login.
- After sign-in, the sidebar shows the real account name/email from the MSAL account.
- API requests carry `Authorization: Bearer <access_token>`, acquired silently for `VITE_API_SCOPE`.
- If MSAL is selected but `VITE_MSAL_CLIENT_ID` / `VITE_MSAL_TENANT_ID` are missing, the app **does not crash**: the login page shows a clear Slovak message (`configError`) and the sign-in button is disabled.

> Auth header logic lives in **one** place: `apiClient.ts` exposes `registerAuthHeaderProvider()`, and each provider registers its strategy (dev headers vs. bearer token). API functions never duplicate header code.

### Routing & guard (`src/App.tsx`)
| Path | Behaviour |
| --- | --- |
| `/` | Redirects to `/chat` when authenticated, otherwise `/login`. |
| `/login` | Login page; redirects to `/chat` if already authenticated. |
| `/chat` | Protected by `ProtectedRoute` — unauthenticated users are sent to `/login`. |
| (loading) | While auth state resolves (MSAL init/redirect), a minimal "Načítava sa…" screen is shown. |

## 5. Environment Configuration

Copy `.env.local.example` to `.env.local` and adjust. All variables are optional (dev defaults are baked in). `.env.local` is git-ignored (`*.local`); **do not commit real secrets**.

| Variable | Mode | Description |
| --- | --- | --- |
| `VITE_AUTH_MODE` | both | `dev` (default) or `msal`. |
| `VITE_DEV_USER_EMAIL` | dev | `X-User-Email` header value. |
| `VITE_DEV_USER_NAME` | dev | `X-User-Name` header value. |
| `VITE_DEV_ENTRA_OBJECT_ID` | dev | `X-Entra-Object-Id` header value. |
| `VITE_DEV_TENANT_ID` | dev | `X-Tenant-Id` header value. |
| `VITE_MSAL_CLIENT_ID` | msal | Azure App Registration (SPA) client id. **Required for msal.** |
| `VITE_MSAL_TENANT_ID` | msal | Entra tenant (directory) id. **Required for msal.** |
| `VITE_MSAL_REDIRECT_URI` | msal | Redirect URI (defaults to `window.location.origin`). |
| `VITE_API_SCOPE` | msal | Backend API scope, e.g. `api://<backend-client-id>/access_as_user`. |

Example `.env.local`:

```env
# Local development (no real identity)
VITE_AUTH_MODE=dev
VITE_DEV_USER_EMAIL=viktor.tindula@company.local
VITE_DEV_USER_NAME=Viktor Tindula
VITE_DEV_ENTRA_OBJECT_ID=local-dev-user
VITE_DEV_TENANT_ID=local-dev-tenant

# For real Microsoft Entra later:
# VITE_AUTH_MODE=msal
# VITE_MSAL_CLIENT_ID=00000000-0000-0000-0000-000000000000
# VITE_MSAL_TENANT_ID=00000000-0000-0000-0000-000000000000
# VITE_MSAL_REDIRECT_URI=http://localhost:5173
# VITE_API_SCOPE=api://BACKEND-APP-CLIENT-ID/access_as_user
```

### What you must configure in Azure / Microsoft Entra (for `msal` mode)
1. **App Registration** for the SPA: type **Single-page application**, with a redirect URI matching `VITE_MSAL_REDIRECT_URI` (e.g. `http://localhost:5173` for dev, plus your prod URL).
2. **Client id** -> `VITE_MSAL_CLIENT_ID`; **Directory (tenant) id** -> `VITE_MSAL_TENANT_ID`.
3. **Backend API registration** exposing an API scope (e.g. `access_as_user`); grant the SPA delegated permission to it. Use the full scope string as `VITE_API_SCOPE` (`api://<backend-client-id>/access_as_user`).
4. The backend must then **validate the Entra JWT** and map claims (`oid`, `tid`, `preferred_username`/`email`, `name`) to the current user. The backend currently resolves identity from dev headers; swapping to JWT validation is the remaining backend task (see `CurrentUserService` TODO).

## 6. Backend Integration & Dev Proxy

- The app calls relative paths under `/api/...`.
- `vite.config.ts` proxies `/api` to `http://localhost:5155` (the `Use.Application.Service` dev URL) with `changeOrigin: true`. Change `server.proxy['/api'].target` to point at a different backend.

### Endpoints used (`src/services/chatApi.ts`)

| Function | Request | Notes |
| --- | --- | --- |
| `askChat(req, signal?)` | `POST /api/chat` | Runs RAG; creates or continues a session. |
| `getChatSessions(signal?)` | `GET /api/chat/sessions` | Current user's sessions (newest first). |
| `createChatSession(title?, signal?)` | `POST /api/chat/sessions` | Available, but FE uses **lazy creation** on first message. |
| `getChatMessages(id, signal?)` | `GET /api/chat/sessions/{id}/messages` | Messages of an owned session. |
| `rateRagQueryLog(id, rating, feedback?, signal?)` | `POST /api/chat/rag-query-logs/{id}/rating` | Store a thumbs up/down (`1` / `-1`, `null` clears). |

> There is **no `/api/users/me` endpoint**, so profile data comes from the auth account (MSAL) or the dev identity — there is no `getCurrentUser()`.

### Request / response shapes

```jsonc
// Request  POST /api/chat
{ "question": "Aký parameter určuje URL Humanet REST API?", "chatSessionId": null }

// Response 200 — chat metadata fields are null when Postgres is disabled
{
  "answer": "markdown string",
  "sources": [{ "title": "...", "url": "...", "sourceSystem": "...", "chunkId": "...", "score": 0.83 }],
  "retrievedChunks": [ /* raw chunks, not rendered in the UI today */ ],
  "chatSessionId": "guid|null",
  "userMessageId": "guid|null",
  "assistantMessageId": "guid|null",
  "ragQueryLogId": "guid|null"
}

// Error (any non-2xx)
{ "error": "human readable message" }
```

The backend returns **`sources`** (not `citations`). `ChatLayout.mapSourceToCitation()` maps each `SourceReference` to the UI citation `{ label, url }`, combining `sourceSystem` and `title`.

`apiClient.ts` centralises fetch: it attaches auth headers, adds `Content-Type: application/json` for bodies, parses `{ error }` bodies into an `ApiError` (carrying the HTTP `status` so the UI can react to 400/401/403/404/502/503), and handles `204 No Content`.

## 7. Key Modules

### `src/types/chat.ts`
Mirrors the backend DTOs (camelCase JSON): `ChatRequest`, `ChatResponse`, `SourceReference`, `RetrievedChunk`, `ChatSession`, `ChatMessage`, `CreateChatSessionRequest`, `RateRagAnswerRequest`, `RatingValue`, `ChatErrorBody`, plus UI view-models `UiCitation` and `CurrentUser`.

### `src/components/ChatLayout.tsx` (the core screen)
Holds conversation + history state with `useState`/`useRef`:
- `messages: UiMessage[]` — current transcript. `UiMessage` carries a local `id`, optional backend `backendId`, `role`, `content`, `citations?`, `durationMs?`, `createdAt?`, `ragQueryLogId?`, `rating?`, `failed?`.
- `sessions`, `activeSessionId`, plus loading/error flags for sessions and messages.
- Separate `AbortController` refs for the chat request, session loading, and message loading.

Behaviour:
- **Sidebar history** loads from `GET /api/chat/sessions` on mount, shows loading/empty/error states, and highlights the active session.
- **Selecting a session** aborts any in-flight chat request, sets `activeSessionId`, and loads its messages via `GET /api/chat/sessions/{id}/messages`.
- **"Nový chat"** cancels in-flight requests and clears the active session/messages/error (no backend session is created until the first message — *lazy creation*).
- **Send flow** (`handleSend`): optimistically appends the user message, calls `askChat({ question, chatSessionId: activeSessionId })`, then appends the assistant message (answer, mapped citations, duration, `ragQueryLogId`). It patches the optimistic user message with the backend `userMessageId`, adopts the returned `chatSessionId`, and refreshes the sidebar. On failure the user message is marked `failed` and the error is surfaced — no fake assistant message is created.
- **Rating**: assistant answers with a `ragQueryLogId` show thumbs up/down. Clicking optimistically sets the rating and calls `rateRagQueryLog`; clicking the same rating again clears it. On API failure the previous rating is restored.
- **Profile area**: shows the real `name` / `email` / `initials` from `useAuth().user` and a logout button (`useAuth().logout`).

### `src/components/ChatInputPanel.tsx`
Reusable controlled input used by both empty and chat states.

## 8. Getting Started

Prerequisites: **Node.js 20+** and npm.

```bash
cd use-fe
npm install        # install dependencies (includes MSAL)
cp .env.local.example .env.local   # optional; dev defaults work without it
npm run dev        # start Vite dev server (default http://localhost:5173)
```

For real answers and history, run the backend `Use.Application.Service` on `http://localhost:5155` (or update the proxy target). History endpoints return `503` when PostgreSQL is disabled — the sidebar shows a friendly Slovak message in that case.

### Available scripts (`package.json`)

| Script | Command | Description |
| --- | --- | --- |
| `npm run dev` | `vite` | Start the dev server with HMR. |
| `npm run build` | `tsc -b && vite build` | Type-check then produce a production build in `dist/`. |
| `npm run preview` | `vite preview` | Serve the production build locally. |
| `npm run lint` | `eslint .` | Lint all source files. |

## 9. Conventions & Notes for Contributors (and LLMs)

- **Language:** User-facing strings are in **Slovak**. Match existing tone/casing when adding copy.
- **Styling:** Inline `style` objects are the pattern. Follow the dark palette (`#212121`, `#171717`, `#2f2f2f` surfaces; `#ececec` text; `#9b9b9b`/`#7d7d7d` muted text).
- **State:** React `useState`/context only — no global store library.
- **Icons:** Inline SVG React components rather than an icon library.
- **Types first:** Update `src/types/chat.ts` when the backend contract changes; `chatApi.ts` and `ChatLayout.tsx` depend on it.
- **Request cancellation:** Preserve the `AbortController` pattern when modifying chat/session/message flows.
- **API layer:** Keep fetch + error handling centralised in `apiClient.ts`; don't duplicate it in feature code.

### Current limitations / known TODOs
1. **Microsoft Entra values** — `msal` mode needs real `VITE_MSAL_CLIENT_ID` / `VITE_MSAL_TENANT_ID` / `VITE_API_SCOPE` and a backend that validates the Entra JWT (see section 5). MSAL is fully wired in code but **not configured**.
2. **Source-scope wiring** — the `KnowledgeBase` / `Slov-lex` checkboxes are UI-only; the backend does not yet support source filtering.
3. **Streaming responses** — answers arrive in one shot; no token-by-token streaming.
4. **Advanced profile menu** — the profile area is a simple display + logout; no dropdown/menu.
5. **Retrieved-chunk UI** — `retrievedChunks` is returned but not rendered.
6. **Source permissions** — not implemented.
7. **Accessibility & responsiveness** — layout uses fixed pixel widths; review for mobile and a11y.
